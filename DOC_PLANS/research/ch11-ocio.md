## Key Concepts to Reuse

### Integration Architecture
- OCIO provides "cross application color consistency" across pipelines
- RV supports "OCIO v2, but is limited to the legacy API of OCIO v1"
- The `ocio_source_setup` package supplements (not replaces) RV's default source_setup
- Permits mixed OCIO and RV-managed imagery in same session
- Color space detection via filename pattern matching

### OCIO Node Types

**OCIOFile**: Input color space conversion (e.g., lg10 cineon to scene-linear)

**OCIODisplay**: Display color correction (scene-linear to viewing transforms)

**OCIOLook**: Creative look modifications with directional control (forward/inverse)

**OCIONode**: Generic implementation for secondary color correction or user-level adjustment

### Node Properties

| Property                  | Function                                      |
|---------------------------|-----------------------------------------------|
| `ocio.function`           | "color", "look", or "display" mode            |
| `ocio.lut`                | Internal storage for generated 3D LUT         |
| `ocio.active`             | Enable/disable node processing                |
| `ocio.lut3DSize`          | LUT resolution (default: 32)                  |
| `ocio.inColorSpace`       | Input color space identifier                  |
| `ocio_color.outColorSpace`| Output space for color function               |
| `ocio_look.look`          | OCIO command string for looks                 |
| `ocio_look.direction`     | Forward (0) or inverse (1) transformation     |
| `ocio_display.display`    | Display name from OCIO configuration          |
| `ocio_display.view`       | View name from OCIO configuration             |

### Configuration
- Requires `OCIO` environment variable set to config file path before launch
- Config files typically facility-specific (e.g., `/OCIO/spi-vfx/config.ocio`)
- Package must be enabled in Preferences, requires restart

### GPU Path
- RV uses "the OCIO GPU path instead of the more common CPU path"
- Creates performance advantages but may have subtle rendering differences vs CPU
- "The color allocation parameters in the config file can have a big effect on the quality of the OCIO LUTs generated by the library"
- Internally generates 3D LUTs from OCIO transforms for GPU application

### Color Space Detection
- Filename pattern matching detects OCIO color spaces
- When OCIO-managed filename detected, OCIO nodes replace standard RV color processing
- Unrecognized files continue using existing RV color management
- Once any OCIO media detected, OCIO display corrections apply to entire session

### Display/View Transforms
- Convert scene-referred linear imagery to monitor-appropriate output
- Multiple display/view combinations configurable per facility
- Users select via OCIO menu interface

### Look Transforms
- Apply creative adjustments via `OCIOLook` node
- Support bidirectional application (forward and inverse)

## What Does NOT Apply to OpenRV Web
- OCIO library integration (no native OCIO in browser)
- `OCIO` environment variable
- Filename-based color space detection (web may use different detection)
- Package system (ocio_source_setup)
- GPU shader generation from OCIO library
- OCIO v1/v2 API distinctions
- Facility-specific config file loading from filesystem

## Adaptation Notes
- OCIO concepts are NOT directly implementable in the web version without a WASM OCIO build or equivalent
- The web version could potentially:
  - Support pre-baked 3D LUTs exported from OCIO configs
  - Implement common transforms (sRGB, Rec.709, ACEScg) natively in shaders
  - Allow users to upload .cube LUT files as OCIO alternatives
- The display/view transform concept maps to the web renderer's output mode / tone mapping
- The property system (ocio.inColorSpace, ocio_display.display/view) could inform a simplified web UI
- If OCIO WASM becomes available, the GPU LUT generation approach (generate 3D LUT, upload as texture) would work well with WebGL2
- The concept of mixed OCIO/non-OCIO sources is worth preserving in documentation even if implementation differs
